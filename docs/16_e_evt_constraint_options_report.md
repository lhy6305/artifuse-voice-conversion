# `e_evt` 约束增强方案调研报告

## 目的
- 基于当前本地设计稿、代码实现和实验结果，整理 `e_evt` 约束增强的候选方案。
- 本报告只做事实收集、优缺点分析和建议，不直接改代码路径。
- 最终采用哪条路线，由用户拍板。

## 先说人话
现在的问题可以用大白话概括成一句：

- 事件路径 `e_evt` 在训练前半段像“老师盯着辅音别跑偏”；
- 但训练后半段，模型越来越会自己走主干，事件路径就没那么被依赖了。

所以我们接下来不是要证明“它有没有接上”，这个已经证明了。  
现在真正要决定的是：

- 要不要专门加约束，让模型后期也继续重视 `e_evt`；
- 还是接受它退成辅助项，只要最终效果还好就行。

## 当前事实
### 1. 设计稿对 `e_evt` 的原始定位
设计稿希望 `e_evt(t)` 负责：
- 摩擦概率
- 闭塞概率
- 爆破概率
- 浊化概率
- 非周期性强度
- 构音部位
- 构音方式

并且明确提出：
- 爆破瞬态
- 强摩擦高频
- 高频乱流细节

这些部分主要靠事件头与声码器建模，而不是靠低频闭环恢复。

### 2. 当前实现的实际状态
当前简化版实现里：
- 模型输入主要是 frame 级：
  - `energy`
  - `abs_mean`
- `event_target` 主要由：
  - `energy`
  - `delta_energy`
  - `zero_cross`
  - 若干阈值/启发式逻辑

组合而来。

当前损失也比较简化：
- `event_loss` 统一是 frame 级 BCE
- 没有 phone/manner 级离散监督
- 没有 class imbalance 处理
- 没有 late-stage 强化策略
- 没有专门防止 `hidden` 直通绕过 `e_evt`

### 3. 这意味着什么
专业说法：
- 当前 `e_evt` 监督信号较粗，而且与主干可观测声学统计存在较强冗余。
- 模型完全可能在 early stage 借助 `e_evt` 对齐结构，但在 late stage 逐渐转向更容易优化的主干路径。

大白话：
- 现在的事件标签太像“从声音大小和变化速度里猜个大概”。
- 模型前期会用它，但后期可能觉得“我自己直接看主干也差不多能做出来”，于是就没那么依赖它了。

## 当前问题拆分
当前 `e_evt` 后期回落，可能来自 3 类原因：

### A. 标签原因
- 事件标签太粗。
- 事件标签和主干声学特征太重合。
- 很多真正重要的辅音差异没有被明确监督出来。

### B. 损失原因
- `event_loss` 权重和训练日程不足以让后期继续保持依赖。
- 没有对稀有事件、burst、fricative 等做额外关注。

### C. 结构原因
- 当前融合里有 `hidden` 直通。
- 模型可以“知道有事件头，但没必要一直依赖它”。

## 候选方案
下面按改动成本从低到高排序。

### 方案 A：只改训练约束，不改主结构
#### 核心做法
- 保持当前模型结构不变。
- 对 `event_loss` 做增强：
  - 提高后期权重
  - 对稀有事件做 class reweight
  - 对 burst / closure / fricative 类事件单独加权
- 增加 late-stage 监控门槛：
  - 若 `zero_e_evt` 敏感度掉到阈值以下，直接标记该实验不通过

#### 优点
- 成本最低。
- 最快验证“是不是纯训练日程问题”。
- 不会一下把现有 large-scale 基线打碎。

#### 缺点
- 如果根本原因是标签太粗或结构可绕过，效果可能有限。
- 可能只是把 `e_evt` loss 拉高，但没有真正让它承担更不可替代的功能。

#### 大白话
- 这相当于先不换机器结构，只是告诉模型：
  - “后面也得继续重视事件这条线，不然实验算没过。”

### 方案 B：改标签与监督定义，但先不大改结构
#### 核心做法
- 保留当前主体结构。
- 把 `event_target` 从“主要靠启发式阈值”升级为更贴近设计稿的监督：
  - 用文本和对齐信息辅助 phone/manner 标签
  - 区分 fricative / closure / burst / voicing / manner / place
  - 把当前纯启发式事件拆成更接近语音学的子项

#### 优点
- 更符合设计稿原意。
- 从监督源头减少“主干和事件标签过度重合”的问题。
- 一旦标签更好，后面的结构和损失分析会更可信。

#### 缺点
- 数据工程明显更重。
- 需要额外对齐或标签生成逻辑。
- 当前 round1 流程里还没有完善的 phone/manner 监督链。

#### 大白话
- 这相当于先把“事件老师”变专业。
- 现在老师太像只看音量和波形变化在打分，后面要让它真的懂“这是摩擦、那是闭塞、这是爆破”。

### 方案 C：改融合结构，减少 `e_evt` 被绕过
#### 核心做法
- 保持标签和损失大体不变。
- 调整当前融合：
  - 降低 `hidden` 直通自由度
  - 让 `e_evt` 主要控制事件敏感子空间
  - 或把部分输出头做成显式事件门控

#### 优点
- 能直接针对“后期被主干绕过”这个核心风险。
- 若设计得当，`e_evt` 的存在感会更强。

#### 缺点
- 结构风险最大。
- 容易把当前已收敛的基线打散。
- 一旦改坏，难判断是结构问题还是训练问题。

#### 大白话
- 这相当于把线路重新接，让模型没法那么轻松绕开事件控制。
- 但这是动手术，不是调旋钮，风险也最大。

### 方案 D：分两段做，先 A 后 B，最后再决定要不要 C
#### 核心做法
1. 先做方案 A
- 快速验证“只靠训练约束增强能不能把 `e_evt` 拉回来”

2. 若效果仍不足，再做方案 B
- 改善标签和监督定义

3. 只有在 A+B 仍不足时，再考虑方案 C
- 把结构改造作为第三层动作，而不是第一反应

#### 优点
- 路径最稳。
- 每一步都更容易归因。
- 更符合当前项目“先验证，再扩改”的节奏。

#### 缺点
- 总周期最长。
- 不能最快得到“结构上最强势”的事件路径。

#### 大白话
- 这就是“先调参数和训练规矩，再换老师，最后才考虑动大手术”。
- 最稳，但也最慢。

## 我当前的建议
### 推荐路线
- 推荐方案 D。

### 理由
1. 当前最便宜、信息量最高的动作是先验证方案 A
- 因为我们还没有证明问题一定在结构。

2. 当前设计稿其实更支持方案 B 的方向
- 设计稿明确提到：
  - 事件监督应结合 phone/manner 标签
- 而我们现在这点还没做到。

3. 现在直接上方案 C 风险太高
- 当前 large-scale 基线刚刚跑稳。
- 直接大改结构，会让归因变差。

## 可供用户决策的选项
请用户在下面三条路线里选一条：

1. 方案 A
- 先只做训练约束增强。
- 优点是快，缺点是可能治标不治本。

2. 方案 B
- 直接优先做事件标签与监督升级。
- 优点是更符合设计稿，缺点是数据工程更重。

3. 方案 D
- 先 A，再视结果决定是否做 B，最后再考虑 C。
- 这是我当前推荐路线。
