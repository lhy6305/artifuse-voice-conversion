# 上下文恢复入口与协作规范

## 读取顺序
每次开始新一轮工作，或历史上下文被清空时，按以下顺序读取：

1. 本文档 `docs/00_context_bootstrap.md`
2. `docs/01_project_overview_and_plan.md`
3. `docs/02_pitfalls_log.md`
4. `initial_design.md`
5. `initial_design_judg.md`

## 强制规范
### 1. 编码规范
- 全部代码、脚本、文档统一使用 UTF-8 无 BOM。
- 读取文本时必须显式指定 UTF-8；命令行读取优先使用支持显式编码的方式。
- 发现历史文件编码不一致时，先记录到踩坑文档，再转换为 UTF-8 无 BOM 后继续修改。

### 2. Python 环境
- 只能使用仓库根目录下的 `python.exe`。
- 禁止调用系统 Python，避免依赖缺失和环境漂移。
- 后续所有 Python 命令默认写成 `.\python.exe ...`。

### 3. 提权执行约束
- 涉及声音处理、Torch 训练、原生库调用、长时间推理、批量特征提取等业务代码时，必须使用沙盒提权运行。
- 普通只读检查、轻量文本处理、文档维护可先在沙盒内执行。
- 如果关键命令因沙盒限制失败，应立即改为提权重试，不绕过该要求。

### 4. 文档维护纪律
- 推进任何任务时，必须同步更新磁盘文档，不能只留在对话里。
- 文档至少持续覆盖以下内容：
  - 项目结构介绍
  - 总任务大纲
  - 当前任务进度
  - 当前阶段验收标准
  - 下一阶段任务
  - 开发踩坑点
- 旧文档若过时，必须修改说明或移入归档目录，禁止直接删除。

### 5. 恢复与交接要求
- 应假设历史对话随时会丢失，所有关键判断、阶段结论、阻塞项都要落盘。
- 进入下一阶段前，先确认当前文档是否足以支持其他人接手。

### 6. 目录结构与临时文件管理
- 无论何时，都必须维护清晰的项目目录结构。
- 临时文件必须放在专门位置，并带有明显标记，例如 `tmp/`、`scratch/`、`*_tmp.*`。
- 临时文件在使用后必须判定去向：
  - 若仍有长期价值，迁移为正式文件并更新文档。
  - 若无价值，及时清理删除。
- 不允许长期把实验输出、草稿脚本、一次性结果混放在根目录。
- 必要时可自行建立 Git 仓库，但仍需保持目录和提交边界清晰。

### 7. 方案决策流程
- 所有方案决策最终都交由用户判断。
- 在给出方案选项前，必须先完成必要的数据收集和事实核对。
- 汇报方案时，必须直接说明：
  - 当前观察到的数据事实
  - 可选方案
  - 每个方案的优点
  - 每个方案的缺点或风险
- 建议选择及其理由
- 未经用户确认，不把存在明显取舍的方案当作既定结论写死。

### 8. 训练执行纪律
- 跑训练时必须打印明确时间信息，至少包含开始时间、结束时间和耗时。
- 在任何大规模训练前，必须先完成小规模训练验证，确认命令、数据读取、日志落盘、checkpoint 和基础速度都正常。
- 禁止在小规模训练尚未跑通前，直接依据主观感觉或单次命令响应时间去估计硬件速度与完整训练耗时。

### 9. 网络使用纪律
- 除了助手内置的搜索能力外，工作区默认全部离线运行。
- 如需使用外部网络，例如：
  - 拉取项目依赖代码
  - 拉取预编译模型
  - 安装 `pip` 包
  - 下载数据或工具链
- 必须先停止当前自动推进，向用户简要说明原因，并直接给出建议命令。
- 这类操作不通过沙盒提权代替，也不由助手擅自联网执行；等待用户手动操作后再继续。

### 10. 沟通可理解性
- 在保持专业严谨技术术语的同时，额外补几句大白话解释当前在做什么。
- 特别是以下场景必须补充通俗说明：
  - 正在执行的步骤
  - 发现的问题
  - 需要用户决策的分歧点
  - 某个结果为什么重要
- 目标是让非实现细节视角下也能看懂“这一步到底在干什么、为什么要做”。

### 11. Git 忽略与恢复边界
- `.gitignore` 的默认目标不是“尽量少提交文件”，而是“在不泄露敏感数据、不把可重建大包体塞满仓库的前提下，最大化意外丢失后的可恢复状态”。
- 允许长期保留在仓库中的内容，默认优先级如下：
  - 代码、配置、规范文档、阶段报告
  - 实验 summary / review / dataset index 等恢复元数据
  - 少量高价值 checkpoint，例如正式训练目录下 `checkpoints/` 的模型状态
- 默认继续忽略的内容：
  - 原始或派生音频
  - 凭证、本地可执行文件、本地环境缓存
  - 可由现有代码和元数据重新导出的 package 级 payload，例如 `reports/runtime/**/packages/**`
- 新增忽略规则时，禁止只按扩展名或目录名一刀切；必须先判断它会不会把以下高恢复价值产物一起误伤：
  - checkpoint
  - dataset index
  - 正式 summary / review
  - 关键训练/评估元数据
- 若某类产物体积较大但恢复价值高，优先采用“保留少量关键件、忽略批量可重建件”的规则，而不是整类全部忽略。
- 每次扩大 `.gitignore` 边界后，必须同时更新正式文档，说明：
  - 为什么要忽略
  - 哪些恢复资产仍被保留
  - 哪些内容丢失后可由现有元数据重建

## 当前约定
- 当前设计基线以 `initial_design.md` 为主，以 `initial_design_judg.md` 的风险提醒作为落地优先级校正。
- 第一优先不是实现“完整 V5.1 全链路”，而是先搭建能验证核心假设的工程骨架与离线 MVP。
- 关键验证顺序必须围绕以下风险展开：
  - 无残差主干能否独立成立
  - 流式前端是否能稳定给出基础控制量
  - 伪标签与置信度机制是否误伤辅音
  - 残差支路是否被严格限制为补偿角色
